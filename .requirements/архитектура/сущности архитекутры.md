## ПОРЯДОК: Этот документ создаётся ПЕРВЫМ

Каталог сущностей содержит пронумерованные элементы для всех трёх слоёв.
**ID-схема по умолчанию:** `Object 1, Object 2, ... Object N`.
**Допускается альтернативная ID-схема** (например `AC-01`, `DB-01`), но если используется альтернативная схема — **все ссылки в документах должны быть в одном формате**.

**Важно:** агент **НЕ придумывает** сущности; он извлекает их из стека/сценариев/требований. Если данных нет — фиксируй `Не определено` и создавай задачу в `.tasks/<id>` для уточнения.

Номера (Object 1, Object 2, ... Object N) используются в описаниях слоёв:

```
docs/requirements/архитектура/сущности.md     <- ПЕРВЫЙ (этот шаблон)
    ↓
docs/requirements/архитектура/описание BL.md  <- ссылается на Object N
docs/requirements/архитектура/описание AL.md  <- ссылается на Object N
docs/requirements/архитектура/описание TL.md  <- ссылается на Object N
```

## Обозначения

- Пометка `(опционально)` — элемент не обязателен для базового сквозного сценария и добавляется только если входит в текущий объём проекта.
- Квадратные скобки `[A + B]` — участники взаимодействия (A и B участвуют совместно, без направления).

**Стили связей в ArchiMate:**
- Serving: провайдер → потребитель (стрелки «вверх»: Technology → Application → Business)
- Used-by: потребитель → провайдер (стрелки «вниз»: Business → Application → Technology)
- Realization: более конкретное → более абстрактное (Artifact → Application Component; Data Object → Business Object)
- Deployment: Artifact → Node
- Triggering: причина → следствие (события/процессы)

Главное — консистентность и корректный тип связи.
