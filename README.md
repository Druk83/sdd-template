# Шаблоны для разработки требований

> **Версия:** 1.2.0

Набор шаблонов для создания технической документации программных проектов с помощью AI-агентов.

## Назначение

Проект решает задачу систематизации процесса создания технической документации:
- Единый формат для всех артефактов
- Последовательность этапов с зависимостями
- Трассируемость от бизнес-требований до тестов
- Поддержка ГОСТ 19 и ГОСТ 34

## Для кого

- **AI-агенты** — используют шаблоны из `.requirements/` для генерации документации
- **Разработчики** — получают структурированную документацию в `docs/requirements/`
- **Аналитики** — имеют единый формат для фиксации требований

## Быстрый старт

### Создание нового проекта

```bash
# Клонировать шаблон
git clone --depth 1 https://github.com/Druk83/sdd-template.git <project-name>
cd <project-name>

# Отвязать от исходного репозитория
rm -rf .git

# Инициализировать свой репозиторий
git init
mkdir -p docs/requirements
git add .
git commit -m "Initial commit from sdd-template"
```

### Запуск документирования

Дать AI-агенту команду:

```
Прочитай .requirements/трек разработки.md и начни документирование с этапа 1.
```

Агент будет:
1. Задавать вопросы о проекте
2. Заполнять шаблоны на основе ответов
3. Сохранять результаты в `docs/requirements/`

## Workflow: разработчик + агент

Полная инструкция: [.manifest/sddmanifest.md](.manifest/sddmanifest.md)

### Краткая последовательность

```
1. Разработчик: "Прими правила, делаем документацию, данные в .source/"
                    ↓
2. Агент: изучает .source/, задаёт уточняющие вопросы
                    ↓
3. Разработчик: отвечает на вопросы
                    ↓
4. Агент: выполняет этап [1], выводит отчёт, ОСТАНАВЛИВАЕТСЯ
                    ↓
5. Разработчик: проверяет, говорит "Продолжай этап [2]"
                    ↓
   ... повторяется для каждого этапа [1]-[9] ...
```

### Ключевые правила

- **Один этап за раз** — агент не выполняет несколько этапов сразу
- **Остановка после этапа** — агент ждёт команды разработчика
- **Уточняющие вопросы** — агент спрашивает, если данных недостаточно
- **Gap tracking** — пробелы фиксируются в `docs/requirements/gap-tracking.md`

### Команды для агента

| Команда | Действие |
|---------|----------|
| `Продолжай этап [N]` | Перейти к следующему этапу |
| `Исправь: <описание>` | Внести правки в текущий артефакт |
| `Покажи статус` | Показать состояние документации |

## Структура проекта

```
.requirements/          # Шаблоны документации (read-only)
  трек разработки.md    # Точка входа, последовательность этапов
  предметная область.md # [1] Глоссарий, акторы, use cases
  обоснование выбора.md # [2] Стек, архитектура, паттерны
  требования/           # [3] ГОСТ 19 / ГОСТ 34
  домены/               # [4] Bounded contexts
  сценарии/             # [5] Event Storming
  структура Данных/     # [6] Схема БД
  архитектура/          # [7] BL / AL / TL
  структура ПО/         # [8] Стек, файловая структура
  тестирование/         # [9] Стратегия тестов

docs/requirements/      # Заполненная документация проекта
.manifest/              # Манифесты процессов (PDD, TDD, Docker)
.tasks/                 # Задачи проекта
.issues/                # Проблемы и баги
.tools/                 # Инструменты автоматизации
```

## Этапы документирования

| Этап | Артефакт | Результат |
|------|----------|-----------|
| [1] | Предметная область | Глоссарий, акторы, UC |
| [2] | Обоснование выбора | Стек, архитектура, методология |
| [3] | Требования | FR-*, NFR-* |
| [4] | Домены | Bounded contexts |
| [5] | Сценарии | События, команды, агрегаты |
| [6] | Структура данных | Таблицы, индексы |
| [7] | Архитектура | BL + AL + TL + связи |
| [8] | Структура ПО | Файлы, зависимости |
| [9] | Тестирование | Стратегия, тест-кейсы |
