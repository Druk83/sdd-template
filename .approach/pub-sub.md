# Pub/Sub — слабая связность между модулями

Подход для слабой связности между модулями через события.

## Назначение

Используй Pub/Sub на этапах [5.2], [7] для слабой связности между модулями через события.
Критерий выбора: требуется слабая связность между модулями (см. 3.5.3).

## Глоссарий

| Термин | Определение |
|--------|-------------|
| Событие | Domain Event (см. `.approach/event-storming.md`) |

## Процедура (императив)

### Фаза 1: Принятие решения
1. Проверь критерии в `docs/requirements/обоснование выбора.md`, раздел 3.5.3.
2. Зафиксируй решение "Pub/Sub: да/нет" и обоснование в разделе 3.5.3.

### Фаза 2: Перечень событий
1. Используй каталог мероприятий и результаты Event Storming.
2. Определи события, применяемые для связей между модулями.
3. Зафиксируй перечень в `docs/requirements/сценарии/<domain>/каталог мероприятий.md`.

### Фаза 3: Отражение в архитектуре
1. Зафиксируй связи между модулями через события в `docs/requirements/архитектура/описание AL.md`.
2. Проверь согласованность с выбранной архитектурой (раздел 3.5.4).

### Фаза 4: Проверка
1. Решение Pub/Sub отражено в разделе 3.5.3.
2. Перечень событий зафиксирован в каталоге мероприятий.
3. Связи между модулями описаны в архитектуре.

## Артефакты

| Артефакт | Расположение |

## Критерии готовности

1. Решение по Pub/Sub зафиксировано в разделе 3.5.3.
2. События для связей между модулями определены и отражены в каталоге мероприятий.
3. Связи между модулями отражены в архитектурном описании.
